# NotionNext 开放生态长期计划

本文用于说明 NotionNext 的长期演进方向，帮助使用者、开发者、维护者和贡献者理解项目接下来为什么做、优先做什么、哪些能力适合进入开源主线，哪些能力更适合作为外部适配、社区模板或独立服务存在。

这不是商业化计划，也不是某个大版本的承诺清单。它的核心目标是让 NotionNext 在保持开源、自托管、可迁移、可定制的前提下，逐步覆盖更多真实站点场景。

## 长期定位

NotionNext 的长期定位是：

> 用 Notion 维护内容，用独立站完成发布、检索、展示和长期运营。

NotionNext 仍然会重视个人博客，这是项目最重要、最稳定的使用场景之一。同时，很多站长已经在用 NotionNext 搭建知识库、产品官网、导航站、课程站、社区官网和项目文档站。长期计划要承认这些真实用法，并把可复用的能力沉淀下来。

项目不应把所有复杂能力都塞进核心代码。更健康的方式是：核心保持稳定，接口保持开放，场景通过模板和文档逐步扩展，外部系统通过适配器接入。

## 读者视角

这份计划同时面向两类读者：

- **使用者和站长**：可以用它判断 NotionNext 是否适合自己的站点，以及未来哪些场景会得到更好支持。
- **开发者和贡献者**：可以用它判断哪些改动更容易被主线接受，以及如何把功能拆成文档、模板、接口和实现。

无论是哪类读者，都不需要依赖任何特定服务才能使用 NotionNext。自托管、Fork、二次开发和迁移能力应始终保留。

## 核心原则

1. **开放优先**
   部署、主题、SEO、搜索、评论、统计、配置、文档、数据读取、升级体验等共性能力，应尽量保持开源、可自托管、可审计。

2. **场景先行**
   新方向先通过文档、Notion 模板、主题配置和示例站验证。只有被多个场景反复使用的能力，才适合抽象进核心。

3. **接口清晰**
   登录、表单、订阅、会员、线索、支付、CRM、通知等能力差异很大，不应一次性内置成一个重系统。更适合先定义字段约定、配置约定和适配器接口。

4. **可替换**
   外部服务只能作为选项。项目应避免把站点绑定到某个托管平台、评论系统、统计系统、表单系统、支付系统或邮件系统。

5. **不增加普通用户负担**
   新能力不能让普通博客用户的部署、配置和升级变复杂。复杂场景可以通过模板、插件和文档逐步启用。

6. **社区贡献友好**
   文档、模板、主题、测试、案例、问题复现和使用反馈都应被视为有效贡献，不应只重视核心代码贡献。

## 四条开放方向

### 1. 站点运营能力

目标：让 NotionNext 不只展示内容，也能帮助站长完成订阅、反馈、互动、线索收集和内容分层等轻量运营任务。

适合进入开源主线的部分：

- 表单、订阅、登录状态、内容可见性等扩展接口。
- Notion 数据库字段约定和示例模板。
- 基础表单提交示例。
- 登录可见内容的最小演示。
- 评论、统计、搜索、广告、邮件订阅等现有能力的文档索引。

适合作为外部适配或可选扩展的部分：

- 写入队列。
- 邮件通知。
- 用户资料页。
- 支付回调。
- 用户标签和分组。
- 运营看板。
- 多站点管理。

这个方向不等于要把 NotionNext 做成完整会员平台。短期目标是先补齐站点运营的基础接口和文档，让不同站长可以按自己的需求接入外部系统。

### 2. 部署与维护支持

目标：降低部署、升级、配置和排查问题的门槛，让更多非专业开发者也能长期维护自己的站点。

开源主线应加强：

- 部署前配置检查。
- 构建和缓存诊断。
- 常见平台部署文档。
- 升级指南。
- 主题兼容性检查。
- 更可读的失败信息。
- 常见问题排查手册。

可选支持可以承接：

- 部署协助。
- 域名和 HTTPS 配置协助。
- 升级协助。
- 主题安装协助。
- SEO 基础配置协助。
- 备份和恢复建议。

这些支持不应成为使用 NotionNext 的前提。它们只是为不想花时间排查基础问题的用户提供额外选择。

### 3. 主题与模板生态

目标：把 NotionNext 的主题优势扩展为可复用的场景方案，让用户可以从“我想做什么站”出发，而不是先理解所有技术细节。

一个完整模板不只包含页面样式，还可以包含：

- Notion 数据库结构。
- 推荐字段。
- 主题配置。
- 示例内容。
- 部署说明。
- 可选插件组合。
- 常见修改位置。

优先场景：

- 个人博客。
- 知识库。
- 开源项目官网。
- 产品官网。
- 课程站。
- AI 工具导航站。
- 社区官网。

推进方式：

- 先建立模板提交规范。
- 再维护精选模板目录。
- 鼓励社区模板、官方示例模板和第三方模板共存。
- 对模板质量、可维护性和兼容性建立基本检查。

模板生态的重点是开放复用，而不是制造封闭入口。用户应能清楚知道模板改了哪些字段、依赖哪些配置、如何迁移和二次修改。

### 4. 场景模板：AI 工具导航站

目标：把 AI 工具导航站作为一个具体场景模板来验证 NotionNext 的内容组织能力，而不是一开始做成独立平台。

选择这个场景的原因：

- 与 Notion 数据库天然匹配。
- 适合展示筛选、标签、榜单、分类、详情页和 SEO 页面。
- 可以验证模板规范、主题组件、提交入口和站点运营能力。
- 复杂度适中，便于拆成多个小 PR。

第一阶段只做模板和文档：

- `Tool` 数据库字段规范。
- 工具列表页。
- 分类页。
- 标签页。
- 工具详情页。
- 推荐榜单。
- 提交入口。
- SEO 元数据规范。

如果该模板产生真实使用案例，再把可复用能力抽象为通用模块。

## 阶段计划

### 0-3 个月：统一叙事和基础入口

目标：让社区理解 NotionNext 是 Notion 驱动的独立站框架，既适合博客，也适合更多内容站场景。

重点任务：

- 完善愿景、路线图和场景文档。
- 梳理博客、知识库、导航站、产品官网、社区官网的推荐路径。
- 建立模板文档结构。
- 为 AI 工具导航站写第一版模板说明。
- 梳理现有邮件订阅、评论、统计、搜索、广告等运营能力。
- 补充部署检查和升级排查文档。

验收标准：

- 新用户能判断自己适合哪类站点。
- 贡献者能看到可拆分的小任务。
- 站长能按场景找到模板和配置说明。

### 3-6 个月：做出第一个完整场景模板

目标：让一个具体场景跑通内容、模板、部署和基础互动。

优先建议选择 AI 工具导航站或课程知识库。

重点任务：

- 定义场景数据库字段。
- 提供示例 Notion 模板。
- 提供主题配置示例。
- 支持提交入口或外部表单链接。
- 支持分类、标签、筛选和详情页的稳定展示。
- 形成一篇完整的从复制模板到上线的教程。

验收标准：

- 一个普通站长可以按教程搭建完整站点。
- 模板能沉淀真实案例和反馈。
- 不需要改核心代码也能完成大部分配置。

### 6-12 个月：抽象扩展接口

目标：把前期场景中反复出现的能力抽象成稳定接口。

重点任务：

- 表单提交适配器。
- Notion 数据写入规范。
- 登录状态接口。
- 内容可见性规则。
- 邮件订阅适配器。
- 用户资料字段。
- 站点事件记录规范。

验收标准：

- 扩展能力不与某个主题硬绑定。
- 普通站点可以不用这些能力。
- 复杂站点可以通过配置或插件接入。
- 外部系统可以被替换。

### 12 个月以上：形成开放生态

目标：让开源项目、模板生态、外部适配和可选支持互相增强。

重点任务：

- 模板目录和推荐标准。
- 主题质量检查。
- 外部适配器清单。
- 可选支持边界说明。
- 多站点管理和升级策略调研。
- 更多真实站点案例。

验收标准：

- 开源仓库仍然可自托管、可迁移、可二次开发。
- 外部服务不会阻塞开源使用。
- 模板和真实案例能持续反哺主线能力。

## 不做什么

短期内不建议做：

- 完整会员 SaaS 平台。
- 与 Ghost、Substack、Patreon 正面竞争的订阅平台。
- 强绑定某个托管服务的闭源内核。
- 把支付、权限、CRM、邮件全部塞进主仓库。
- 一次性重写主题系统。
- 为单个站点加入不可复用的定制逻辑。
- 让基础使用路径依赖某个付费服务。

## 维护者判断清单

当一个新功能被提出时，先问：

1. 它是多数 NotionNext 站长的共性问题吗？
2. 它能否先用文档、模板或配置解决？
3. 它是否需要进入核心，还是适合作为适配器？
4. 它是否会增加普通博客用户的维护负担？
5. 它是否能沉淀为可复用的主题、组件或数据契约？
6. 它是否会破坏自托管、可迁移和开源透明性？
7. 它是否允许用户替换外部服务？

只有当这些问题有清晰答案时，才适合进入实现阶段。

## 推荐的下一批小任务

| 方向 | 最小任务 |
| --- | --- |
| 场景文档 | 新增“如何选择站点类型”文档，覆盖博客、知识库、导航站、课程站、产品官网 |
| 模板生态 | 定义模板包结构：Notion 模板、主题配置、示例内容、部署说明 |
| AI 导航站 | 设计 `Tool` 数据库字段，并提供第一版模板说明 |
| 运营能力 | 梳理现有评论、统计、搜索、邮件订阅、广告能力，形成运营能力索引 |
| 部署维护 | 新增部署检查清单和常见升级问题排查页 |
| 社区路线 | 把 `Member` / `Event` 与社区官网路线图拆成可执行 Issue |

这份计划的关键不是同时启动所有方向，而是让每个方向都能从开源主线自然长出来：先文档和模板，再接口和示例，最后才是更复杂的外部适配和可选支持。
